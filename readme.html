Overview

SecurePay Banking Terminal

A full-stack banking web application demonstrating defence-in-depth security practices. Built with PHP 8.2, MySQL 8, Apache 2.4. Deployed via Docker Compose. Themed after Indian ATM interfaces of the early 2000s — SBI, PNB era.

Every user-facing feature has a corresponding security control. This document covers architecture, file reference, security decisions, test procedures, and the full attack surface.

Implemented Features
  • User registration with bcrypt (cost 12) password hashing
  • Login with per-IP rate limiting and timing-safe comparison
  • Secure session management — httponly, samesite=Strict, regeneration on login
  • CSRF tokens on every state-changing form including logout
  • Profile editing — name, bio, email, avatar
  • Secure image upload — MIME validation, outside webroot, random filename
  • User search by username — includes own account, no ID exposure
  • Money transfers by username — internal IDs never touch the browser
  • Atomic transfers with row-level locking — no race conditions
  • Transaction history with pagination
  • Activity logging to file + database on every page load
  • Admin panel — log viewer, filtering by user/IP/date
  • Docker deployment with dropped capabilities and no exposed DB port
  • Random non-sequential public user IDs (12-char hex)
  • Password visibility toggle on all password fields
  • Custom confirm modal — no browser alert() or confirm() dialogs
  • POST-based logout with CSRF token validation
  • Root redirect — visiting / goes to login or dashboard

Security Architecture

Authentication

ThreatControl
Brute force5 attempts/IP/15 min → DB lockout
Timing-based username enumerationDummy hash compared on unknown user
Message-based username enumerationSingle "Invalid credentials" message
Session fixationsession_regenerate_id(true) on login
Session cookie theft via XSShttponly=1, samesite=Strict
Stale session after logoutThree-step destroy: clear array + delete file + expire cookie
Cached page after logoutCache-Control: no-store on all PHP
CSRF-forced logoutLogout is POST-only with CSRF token
Logout via URL barGET to logout.php is silently ignored

User ID Design

OldProblemNew
Sequential integer (1, 2, 3…) Attacker enumerates all accounts by incrementing bin2hex(random_bytes(6)) = 12-char hex

Internal id (INT) is used only for FK relationships — stays on the server. public_id is used in all URLs. Transfers use username — no ID ever enters the browser.

Money Transfer Security

StepControl
Sender identityFrom $_SESSION['user_id'] — never from POST
Recipient lookupBy username server-side — internal IDs never sent to browser
AtomicityBEGIN TRANSACTION wraps deduct + credit + log
Race condition / double-spendSELECT ... FOR UPDATE row lock on sender
Insufficient fundsUPDATE WHERE balance >= amount — atomic check
Self-transferRejected explicitly before transaction begins
Float errorsround($amount, 2) + DECIMAL(12,2) in DB

Image Upload (5 Layers)

LayerCheck
1$_FILES['error'] === UPLOAD_ERR_OK
2filesize($tmp_name) ≤ 2MB — not $_FILES['size'] which can be forged
3finfo(FILEINFO_MIME_TYPE) reads magic bytes — not filename/extension
4Extension whitelisted against detected MIME type
5getimagesize() — file must parse as a valid image

Stored in /var/www/private/uploads/ — outside Apache webroot. Random filename: bin2hex(random_bytes(16)) . '.' . $ext. User's original filename is discarded.

HTTP Headers

X-Content-Type-Options:  nosniff
X-Frame-Options:         DENY
X-XSS-Protection:        1; mode=block
Referrer-Policy:         strict-origin-when-cross-origin
Content-Security-Policy: default-src 'self'; object-src 'none'; frame-ancestors 'none'
Cache-Control:           no-store, no-cache, must-revalidate  (all PHP/HTML)

Set in PHP (security_headers.php) and duplicated at Apache level in vhost.conf. Two layers: if PHP crashes before headers are sent, Apache still sends them.

Apache — vhost.conf

DirectoryIndex login.php         # / → login
Options -Indexes -FollowSymLinks # no directory listings, no symlink escapes
AllowOverride None               # on config/ and includes/ (explicitly)
Require all denied               # on config/, includes/, /var/www/private

PHP — security.ini

expose_php         = Off
display_errors     = Off
allow_url_fopen    = Off
allow_url_include  = Off
open_basedir       = /var/www/html:/var/www/private:/tmp
disable_functions  = exec,passthru,shell_exec,system,proc_open,popen,...

2. First Run

# or:
docker compose up --build -d

3. Schema Upgrade

If upgrading from a previous version — the schema now has a public_id column. You must wipe the old volume:

docker compose down -v   # WARNING: deletes all data
docker compose up --build -d

4. Create Admin Account

docker exec securepay_app php -r \
  "echo password_hash('YourAdminPass!', PASSWORD_BCRYPT, ['cost'=>12]);"

docker exec -it securepay_db mysql -u root -p
# USE securepay;
# INSERT INTO admins (username, password_hash) VALUES ('admin', '<hash>');

Test Checklist
Registration
Register with valid data → Rs. 100 credited, redirected to login
Register duplicate username → generic "already exists" (no info leak)
Weak password (short / no uppercase / no number) → rejected
Mismatched passwords → rejected
Check DB: public_id is 12-char hex, not sequential integer
Login
Correct credentials → dashboard
Wrong password → "Invalid credentials" (no field distinction)
Unknown username → same message, same response time
5 failures from same IP → 15-min lockout
Correct credentials during lockout still blocked
Password Visibility
Eye button toggles type between password/text on login page
Eye button works on both password fields on register page
Logout
LOGOUT button works — session destroyed, redirected to login
Back button after logout → login page (cache headers working)
Visit /logout.php in URL bar → redirected, NOT logged out
Exactly one LOGOUT button visible in nav
Root Redirect
Visit http://localhost:8080/ unauthenticated → login page
Visit http://localhost:8080/ authenticated → dashboard
Profile and Image Upload
Edit name and bio → saved correctly
Upload valid JPG → image displays on profile
Upload PHP file → rejected at MIME check
Upload file > 2MB → rejected at size check
Upload PHP file renamed to .jpg → rejected (magic bytes check)
Search
Search own username → appears in results marked "(you)"
Own account shows VIEW only — no SEND button
Another user shows VIEW and SEND buttons
SEND prefills recipient username in transfer form
Transfers
Transfer to valid username → balances updated, transaction recorded
Amount exceeds balance → "Insufficient balance" error
Transfer to self → rejected
Transfer to nonexistent username → "User not found"
REVIEW TRANSFER shows modal (not browser alert)
CANCEL in modal → form not submitted
Security
Visit /dashboard.php without login → redirect to login
Visit /config/db.php directly → 403 Forbidden
Visit /includes/csrf.php directly → 403 Forbidden
POST transfer without CSRF token → 403
POST logout without CSRF token → 403
Response headers: X-Frame-Options DENY, nosniff present
Admin
Login at /admin_login.php → logs panel
Filter by username, IP, date → results update
Wrong admin password 3 times → admin lockout
Visit /admin_logs.php without admin session → redirected
